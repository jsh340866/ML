03. 풀스택 — 로그인하고 화면에서 예측하기

인증이 완성된 풀스택 템플릿에 ML을 연결한다. 무엇을 추가·수정하는지 순서대로 짚은 뒤 실행한다.

대상: 비전공 초보 · 소요: 약 3시간 · 선수: 01·02 완료 · 필요: Docker Desktop 실행 중

이 문서는 두 부분입니다 — 1부 코드 작성 순서(템플릿에 무엇을 추가·수정하는가, 5단계) → 2부 실행 실습(띄우고 화면에서 예측). 이미 있는 인증 코드는 건드리지 않습니다.

점선 밑줄 용어에 마우스를 올리면 툴팁 설명이 나옵니다. 각 단계는 코드 → 수행결과 → 결과 설명 순서입니다.

이 문서의 결과 표기

초록색 수행결과는 실제로 확인된 출력입니다. 주황색 예상 출력은 도커 환경·시각·컨테이너 ID에 따라 달라지는 부분이라 형태만 보여 준 것이니, 글자가 완전히 같지 않아도 정상입니다.

핵심 개념 — 왜 서비스를 나누나

설명

한 덩어리로 만들면 편할 텐데 왜 나눌까요? 각자 잘하는 게 다르기 때문입니다.

서비스기술맡은 일포트

핵심

Java는 파이썬 pkl을 읽지 못합니다. 그래서 머신러닝은 파이썬(ML)이 맡고, 로그인·회원관리는 Java(BN)가 맡고, 둘은 HTTP로 대화합니다.

데이터 흐름: 브라우저(FN) → BN(인증 확인 후 중계) → ML(추론) → model.pkl

실무의 마이크로서비스 구조를 5개 컨테이너로 축소해 놓은 것이 이 실습입니다.

설명

BN에게 ML의 주소를 알려주고, ML이 뜬 다음 BN이 뜨게 순서를 잡는다.

코드 설명

ML_URL: http://ml-container:8002: localhost가 아니라 컨테이너 이름입니다. 컨테이너 입장에서 localhost는 "자기 자신"이라 ML을 못 찾습니다. 가장 자주 하는 실수입니다.

포트도 8002(컨테이너 내부 포트)입니다. 호스트 매핑(8002:8002)과 무관하게 내부에서는 원래 포트로 부릅니다.

depends_on: ml: ML이 시작된 뒤 BN이 뜹니다.

이 환경변수를 Java가 어떻게 읽는지는 3단계에서 나옵니다.

3BN — PredictionController.java 신규 작성

설명

BN은 예측을 직접 하지 않는다. 요청을 ML로 넘기고 결과를 그대로 전달하는 중계(프록시) 역할만 한다.

3-1. 클래스 뼈대 & ML 주소 주입

@RestController
@Slf4j
@CrossOrigin(origins = "*")
public class PredictionController {

    // docker-compose 의 BN 환경변수 ML_URL 로 주입 (기본값: ml-container:8002)
    @Value("${ml.url:http://ml-container:8002}")
    private String mlUrl;

    private final RestTemplate rt = new RestTemplate();
코드 설명

@Value("${ml.url:...}")가 2단계의 ML_URL을 읽습니다. ML_URL → ml.url 로 자동 매칭됩니다.

:http://ml-container:8002 뒤쪽은 기본값입니다. 환경변수가 없어도 동작합니다.

RestTemplate: BN이 ML에게 요청을 보낼 때 쓰는 도구입니다.

3-2. 공통 전달 함수 — forward()

설명

7개 엔드포인트가 모두 같은 일(ML로 넘기고 응답 전달)을 한다. 한 함수로 모은다.

    /** ML 로 요청을 넘기고 응답(상태코드 포함)을 그대로 돌려준다 */
    private ResponseEntity<String> forward(HttpMethod method, String path, Object body) {
        HttpHeaders headers = new HttpHeaders();
        headers.setContentType(MediaType.APPLICATION_JSON);
        HttpEntity<Object> request = new HttpEntity<>(body, headers);
        try {
            return rt.exchange(mlUrl + path, method, request, String.class);
        } catch (HttpStatusCodeException e) {
            // ML 이 404 를 주면 BN 도 404 를 준다 (500 으로 뭉개지 않는다)
            log.warn("[ML {}] {} -> {}", method, path, e.getStatusCode());
            return ResponseEntity.status(e.getStatusCode())
                    .contentType(MediaType.APPLICATION_JSON)
                    .body(e.getResponseBodyAsString());
        }
    }
코드 설명 — try/catch가 핵심

rt.exchange(...)는 ML이 404를 주면 예외를 던집니다. 이걸 잡지 않으면 Spring이 500 Internal Server Error로 바꿔 버려 화면에서 원인을 알 수 없게 됩니다.

catch (HttpStatusCodeException e)로 잡아 ML의 상태코드와 본문을 그대로 전달합니다. 02에서 만든 {"detail":"해당 예측 이력이 없습니다"}가 화면까지 살아서 갑니다.

응답 타입이 String인 이유: BN은 내용을 해석할 필요가 없습니다. 받은 JSON을 그대로 흘려보냅니다.

3-3. 요청 형태 맞추기 — wrap()

    /** FN 은 features 만 보낸다 -> ML 이 요구하는 {"features": {...}} 형태로 감싼다 */
    private Map<String, Object> wrap(Map<String, Object> features) {
        Map<String, Object> body = new HashMap<>();
        body.put("features", features);
        return body;
    }
코드 설명

02의 PredictRequest는 {"features": {...}}를 요구합니다(02의 4단계). 그런데 FN은 입력값만 보내는 게 자연스럽죠. 그 차이를 BN이 흡수합니다.

이렇게 하면 화면 코드가 ML의 내부 형식을 몰라도 됩니다.

3-4. 엔드포인트 7개 연결

    @GetMapping("/api/spec")
    public ResponseEntity<String> spec() {
        return forward(HttpMethod.GET, "/spec", null);
    }

    @PostMapping("/api/predict")
    public ResponseEntity<String> predict(@RequestBody Map<String, Object> features) {
        log.info("[predict] features={}", features);
        return forward(HttpMethod.POST, "/predict", wrap(features));
    }

    // ---------------- 예측 이력 CRUD ----------------

    /** C : 예측 후 이력 저장 */
    @PostMapping("/api/predictions")
    public ResponseEntity<String> create(@RequestBody Map<String, Object> features) {
        return forward(HttpMethod.POST, "/predictions", wrap(features));
    }

    /** R : 이력 목록 */
    @GetMapping("/api/predictions")
    public ResponseEntity<String> list(@RequestParam(defaultValue = "50") int limit) {
        return forward(HttpMethod.GET, "/predictions?limit=" + limit, null);
    }

    /** R : 이력 단건 */
    @GetMapping("/api/predictions/{id}")
    public ResponseEntity<String> get(@PathVariable int id) {
        return forward(HttpMethod.GET, "/predictions/" + id, null);
    }

    /** U : 입력을 바꿔 다시 예측하고 갱신 */
    @PutMapping("/api/predictions/{id}")
    public ResponseEntity<String> update(@PathVariable int id, @RequestBody Map<String, Object> features) {
        return forward(HttpMethod.PUT, "/predictions/" + id, wrap(features));
    }

    /** D : 이력 삭제 */
    @DeleteMapping("/api/predictions/{id}")
    public ResponseEntity<String> delete(@PathVariable int id) {
        return forward(HttpMethod.DELETE, "/predictions/" + id, null);
    }
코드 설명

각 메서드가 한 줄입니다. forward와 wrap을 만들어 뒀기 때문입니다. BN 경로(/api/...)와 ML 경로(...)가 1:1로 대응합니다.

/api/ 접두사를 붙인 이유: 템플릿의 기존 경로(/join·/login·/user)와 겹치지 않게 하기 위해서입니다.

중요 — 인증: 이 경로들은 SecurityConfig의 anyRequest().authenticated() 대상입니다. 즉 별도 코드 없이 자동으로 "로그인해야 호출 가능"이 됩니다.

4FN — Predict.jsx 신규 작성

설명

화면을 만든다. 입력 폼을 하드코딩하지 않고 /api/spec 을 보고 자동으로 그린다.

4-1. 상태와 초기 로딩

import { useState, useEffect } from "react";
import api from "../api/axiosConfig";

const Predict = () => {
    const [spec, setSpec] = useState(null);
    const [vals, setVals] = useState({});
    const [rows, setRows] = useState([]);
    const [editId, setEditId] = useState(null);   // null 이면 신규, 값이 있으면 수정 중
    const [result, setResult] = useState(null);
    const [error, setError] = useState(null);

    const initVals = (s) => {
        const v = {};
        s.features.forEach((f) => { v[f.name] = f.kind === "num" ? f.default : 0; });
        return v;
    };

    // R : 이력 목록
    const loadRows = () => {
        api.get("/api/predictions")
            .then((r) => setRows(r.data))
            .catch(() => setError("이력을 불러오지 못했습니다."));
    };

    // 입력 정의를 받아 폼 초기값 구성
    useEffect(() => {
        api.get("/api/spec")
            .then((r) => {
                setSpec(r.data);
                setVals(initVals(r.data));
                loadRows();
            })
            .catch(() => setError("모델 정보를 불러오지 못했습니다. 로그인 상태를 확인하세요."));
    }, []);
코드 설명

useEffect(..., []): 화면이 열리면 /api/spec을 부릅니다.

initVals: spec의 default(수치)와 0(선택 첫 항목)으로 폼 초기값을 만듭니다. 01의 8단계에서 저장한 default가 여기서 쓰입니다.

api: 템플릿의 axiosConfig. 인증 토큰을 자동으로 실어 보냅니다. 그래서 우리가 인증 코드를 안 써도 됩니다.

catch: 로그인 안 했으면 401이 오므로 안내 문구를 띄웁니다.

4-2. C와 U — 하나의 함수로

    // C(신규) 또는 U(수정)
    const onSubmit = async (e) => {
        e.preventDefault();
        setError(null);
        try {
            const r = editId === null
                ? await api.post("/api/predictions", vals)            // C
                : await api.put(`/api/predictions/${editId}`, vals);  // U
            setResult(r.data);
            setEditId(null);
            setVals(initVals(spec));
            loadRows();
        } catch (err) {
            setError("예측 요청에 실패했습니다.");
        }
    };

    // 수정 시작 : 이력의 입력값을 폼에 채운다
    const onEdit = (row) => {
        setVals(row.features);
        setEditId(row.id);
        setResult(null);
    };

    // D : 삭제
    const onDelete = async (id) => {
        try {
            await api.delete(`/api/predictions/${id}`);
            if (editId === id) onCancel();
            loadRows();
        } catch (err) {
            setError("삭제에 실패했습니다.");
        }
    };
코드 설명

editId 하나로 신규/수정을 구분합니다. null이면 POST(C), 값이 있으면 PUT(U). 폼을 두 개 만들 필요가 없습니다.

저장 후 반드시 loadRows(): 서버의 최신 목록을 다시 받아옵니다. 화면에서 임의로 배열을 조작하면 서버와 어긋날 수 있습니다.

onEdit이 row.features를 그대로 폼에 넣습니다 — 02가 저장해 둔 입력이 되살아납니다.

코드 설명 — 이 실습 구조의 핵심

spec.features.map(...): 입력칸을 코드에 적어두지 않고 spec을 보고 그립니다.

그래서 01에서 피처를 바꾸고 다시 학습하면 이 화면이 저절로 바뀝니다. React 코드는 한 줄도 안 고쳐도 됩니다.

      </Routes>
    </Router>
  );
}

참고 — BN이 제공하는 API

메서드BN 경로설명ML로 전달
GET/api/spec입력 정의(폼 자동 생성)/spec
POST/api/predict예측만 (저장 안 함)/predict
POST/api/predictionsC 예측 후 이력 저장/predictions
GET/api/predictionsR 이력 목록/predictions
GET/api/predictions/{id}R 이력 단건/predictions/{id}
PUT/api/predictions/{id}U 재예측·갱신/predictions/{id}
DELETE/api/predictions/{id}D 이력 삭제/predictions/{id}
BN이 하는 일은 두 가지뿐

1) 형태 맞추기(wrap): FN의 features를 ML이 원하는 {"features": {...}}로 감쌈.

2) 상태코드 전달(forward의 try/catch): ML의 404를 500으로 뭉개지 않고 그대로 전달.

예측 자체는 전혀 하지 않습니다. 그래서 모델을 바꿔도 BN 코드는 안 고쳐도 됩니다.
