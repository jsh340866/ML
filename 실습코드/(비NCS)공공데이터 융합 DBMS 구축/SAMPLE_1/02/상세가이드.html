02. ML API — model.pkl 을 서비스하기

01에서 학습한 모델을 FastAPI로 감싸 REST API로 만든다. 코드를 순서대로 직접 작성한 뒤, 실행해서 호출해 본다.

대상: 비전공 초보 · 소요: 약 3시간 · 선수: 01 완료(model.pkl 필요) · 포트: 8001

이 문서는 두 부분입니다 — 1부 코드 작성 순서(9단계로 main.py·model.py·store.py를 직접 만든다) → 2부 실행 실습(띄우고 호출해 본다). 코드를 먼저 이해하고 실행하세요.

점선 밑줄 용어에 마우스를 올리면 툴팁 설명이 나옵니다. 각 단계는 코드 → 수행결과 → 결과 설명 순서입니다.

핵심 개념 — 왜 모델을 API로 만드나

설명

학습한 모델은 model.pkl이라는 파일일 뿐입니다. 다른 프로그램(웹·앱)이 쓰려면 API로 감싸야 합니다.

03에서 Spring Boot(Java)가 이 모델을 쓰는데, Java는 파이썬 pkl을 직접 못 읽습니다. 그래서 파이썬이 API로 열어 주고, Java는 HTTP로 물어보는 구조를 씁니다.

구분경로차이
예측만/predict결과만 돌려주고 저장 안 함. 03의 BN이 쓰는 경로.
이력 CRUD/predictions예측 결과를 SQLite에 저장하고 조회·수정·삭제.

CRUD는 웹 개발의 기본기입니다. 이 단원에서 ML 예측 결과를 대상으로 CRUD를 직접 만들어 봅니다.

파일 구성 — 무엇을 만들 것인가

설명

01을 먼저 끝내야 합니다. model.pkl과 feature_spec.json은 01이 만들어 준 것이고, 나머지를 우리가 작성합니다.

파일역할누가 만드나
model.pkl학습된 모델 (약 2.4MB)01이 생성
feature_spec.json입력 항목 정의01이 생성
requirements.txt설치할 패키지 목록1단계
model.py모델 로드 · 예측 함수2단계
main.pyFastAPI 엔드포인트 8개3~6, 8단계
store.py예측 이력 저장소(SQLite CRUD)7단계
Dockerfile컨테이너 이미지 정의9단계
predictions.db이력 DB실행하면 자동 생성
작성 순서의 원칙

안쪽부터 바깥쪽으로 만듭니다. 모델 다루기(model.py) → 앱 뼈대(main.py) → 예측 엔드포인트 → 저장소(store.py) → CRUD 엔드포인트 순서입니다.

각 단계마다 바로 실행해서 확인할 수 있게 짰습니다. 한 번에 다 짜고 나서 오류를 찾으면 원인을 알기 어렵습니다.

1부. 코드 작성 순서

1requirements.txt — 무엇을 설치하나

설명

필요한 패키지를 버전까지 고정해 적는다. 버전을 안 적으면 나중에 다른 버전이 깔려 코드가 깨질 수 있다.

# 01(모델 학습) 환경과 같은 버전이어야 model.pkl 을 읽을 수 있다.
fastapi==0.110.0

왜 이것들인가

중요: scikit-learn 버전이 01에서 학습할 때와 다르면 경고가 뜨거나 로드가 실패할 수 있다.

SQLite가 목록에 없는 이유도 알아 두세요.

2model.py — 모델 로드 · 예측 함수

설명

가장 안쪽 부품부터 만든다. API와 무관하게 "pkl을 읽어 예측한다"는 일만 담당한다.

import joblib
import pandas as pd


def load_model(path="model.pkl"):
    """학습된 모델 로드"""
    return joblib.load(path)


def predict(model, input_data):
    """입력 dict -> (예측 클래스, 확률 배열)"""
    df = pd.DataFrame([input_data])
    return model.predict(df)[0], model.predict_proba(df)[0]
코드 설명

pd.DataFrame([input_data]): 딕셔너리 하나를 1행짜리 표로 바꿉니다. 모델은 딕셔너리가 아니라 표를 받습니다.

[0]: 1행만 넣었으니 결과도 1개. 배열에서 첫 번째를 꺼냅니다.

predict와 predict_proba를 둘 다 돌려주는 이유: 화면에 라벨(있음/없음)과 확률을 함께 보여주기 위해서입니다.

주의 — 열 이름과 순서

input_data의 키 이름이 01에서 학습할 때의 열 이름과 같아야 합니다(gu 등). 다르면 오류가 납니다.

3main.py 뼈대 — 모델 로드 & 앱 생성

설명

서버가 뜰 때 딱 한 번 모델과 spec을 읽어 메모리에 올려둔다. 요청마다 읽으면 매번 2.4MB를 로드해 느려진다.

import json
from typing import Dict, Any

from fastapi import FastAPI, HTTPException
from fastapi.middleware.cors import CORSMiddleware
from pydantic import BaseModel

import store
from model import load_model, predict

# 모델 · 입력 정의 로드 (startup 캐싱)
spec = json.load(open("feature_spec.json", encoding="utf-8"))
model = load_model("model.pkl")
LABELS = spec["target_labels"]

app = FastAPI(title=spec["title"] + " ML API",
              description="model.pkl 을 서빙하는 예측 엔드포인트 + 예측 이력 CRUD.")

# React/BN 에서 호출 허용
app.add_middleware(CORSMiddleware, allow_origins=["*"],
                   allow_methods=["*"], allow_headers=["*"])

store.init_db()      # 예측 이력 테이블 준비 (7단계에서 만든다)
코드 설명

모듈 최상단에 두는 이유: 파이썬은 이 파일을 한 번만 실행합니다. 그래서 자연스럽게 "시작할 때 1회 로드"가 됩니다.

LABELS: ["인명피해 없음", "인명피해 있음"]. 모델은 0/1을 주는데 사람에게는 글자로 보여야 하니 번호 → 글자 변환표로 씁니다.

CORS: 03에서 React(3000번)가 이 API(8001번)를 부를 수 있게 허용합니다. 없으면 브라우저가 요청을 차단합니다.

실습용이라 allow_origins=["*"](전부 허용). 실무에서는 허용할 주소를 명시해야 합니다.

4요청 스키마 — 입력 형태 정하기

설명

어떤 JSON을 받을지 Pydantic 클래스로 선언한다. 검증과 문서가 자동으로 따라온다.

class PredictRequest(BaseModel):
    features: Dict[str, Any]   # {"gu":24, "월":3, ...}
이 3줄이 하는 일

1) 형태 강제: 요청 본문이 반드시 {"features": {...}} 여야 합니다. 아니면 FastAPI가 422 오류를 자동으로 돌려줍니다.

2) 문서 자동 생성: Swagger에 입력 예시가 나타납니다.

왜 features로 한 겹 감쌌나? 나중에 {"features": {...}, "memo": "..."} 처럼 항목을 추가하기 쉽기 때문입니다. 03의 Spring Boot가 이 형태에 맞춰 감싸서 보냅니다.

파이썬 딕셔너리를 그냥 return하면 FastAPI가 JSON으로 바꿔 줍니다. 변환 코드를 쓸 필요가 없습니다.

/health는 헬스체크 관행입니다. /spec은 01이 만든 파일을 그대로 돌려주는데, 이 응답 하나가 03의 입력 폼을 자동으로 만듭니다.

6예측 엔드포인트 — run_model() · /predict

설명

예측 로직을 함수로 따로 뺀다. /predict 와 CRUD가 같은 로직을 써야 하기 때문이다.

def run_model(features: Dict[str, Any]) -> Dict[str, Any]:
    """model.pkl 로 예측해 결과 dict 를 만든다 (predict/CRUD 공용)"""
    pred, proba = predict(model, features)
    pred = int(pred)
    return {
        "prediction": pred,
        "label": LABELS[pred],
        "proba": {LABELS[0]: round(float(proba[0]), 4),
                  LABELS[1]: round(float(proba[1]), 4)},
    }


@app.post("/predict")
def do_predict(req: PredictRequest):
    """예측만 한다 (저장 안 함). BN 이 사용하는 엔드포인트."""
    return run_model(req.features)
코드 설명 — 왜 run_model을 따로 뺐나

/predict·POST /predictions·PUT /predictions/{id} 세 곳이 예측을 합니다. 로직을 복사해 두면 나중에 한 곳만 고치고 나머지를 빠뜨리게 됩니다. 한 군데에만 두고 셋이 같이 씁니다.

int(pred)·float(proba[0]): 모델이 주는 값은 numpy 타입이라 JSON으로 못 바꿉니다. 파이썬 기본 타입으로 변환해야 합니다. 실무에서 자주 만나는 오류입니다.

LABELS[pred]: 0 → "인명피해 없음", 1 → "인명피해 있음". 번호를 글자로 바꿔 줍니다.

7store.py — 이력 저장소(SQLite CRUD)

설명

DB를 다루는 코드를 별도 파일로 분리한다. main.py는 "웹 요청 처리", store.py는 "저장" — 역할 분리.

7-1. 연결 & 테이블 생성

import json
import sqlite3
from datetime import datetime

DB_PATH = "predictions.db"


def _con():
    con = sqlite3.connect(DB_PATH)
    con.row_factory = sqlite3.Row      # 결과를 dict 처럼 사용
    return con


def init_db():
    """예측 이력 테이블 생성 (앱 시작 시 1회)"""
    con = _con()
    try:
        con.execute("""
            CREATE TABLE IF NOT EXISTS predictions (
                id         INTEGER PRIMARY KEY AUTOINCREMENT,
                features   TEXT NOT NULL,
                prediction INTEGER NOT NULL,
                label      TEXT NOT NULL,
                proba      TEXT NOT NULL,
                created_at TEXT NOT NULL
            )
        """)
        con.commit()
    finally:
        con.close()
코드 설명

IF NOT EXISTS: 이미 있으면 그냥 넘어갑니다. 그래서 서버를 몇 번 재시작해도 안전합니다.

AUTOINCREMENT: id를 자동으로 매깁니다.

features·proba가 TEXT인 이유: 딕셔너리를 JSON 문자열로 바꿔 저장하기 때문입니다.

try/finally로 반드시 close()합니다. 안 닫으면 연결이 쌓입니다.

7-2. CRUD 함수 4종

# C : 저장하고 저장된 레코드를 돌려준다
def create(features, result):
    con = _con()
    try:
        cur = con.execute(
            "INSERT INTO predictions(features, prediction, label, proba, created_at)"
            " VALUES (?, ?, ?, ?, ?)",
            (json.dumps(features, ensure_ascii=False), result["prediction"], result["label"],
             json.dumps(result["proba"], ensure_ascii=False),
             datetime.now().isoformat(timespec="seconds")),
        )
        con.commit()
        pid = cur.lastrowid
    finally:
        con.close()
    return get(pid)


# R : 단건 (없으면 None) / 목록 (최신순)
def get(pid): ...
def list_all(limit=50): ...        # ORDER BY id DESC LIMIT ?

# U : 다시 예측한 결과로 갱신 (없으면 None)
def update(pid, features, result): ...

# D : 삭제 (지운 게 있으면 True)
def delete(pid): ...
코드 설명

? 자리표시자: 값을 직접 문자열로 붙이지 않고 따로 전달합니다. SQL 인젝션을 막는 기본기입니다.

ensure_ascii=False: 한글이 \uc778\uba85 처럼 깨져 저장되지 않게 합니다.

create가 마지막에 get(pid)를 부르는 이유: DB가 매긴 id와 저장된 실제 값을 그대로 돌려주기 위해서입니다.

update·delete는 rowcount로 "그 id가 있었는지"를 판단합니다. 이게 8단계 404 처리의 근거입니다.

8CRUD 엔드포인트 — main.py에 5개 추가

설명

7단계에서 만든 store 함수를 URL에 연결한다. 엔드포인트는 얇게 — 실제 일은 store와 run_model이 한다.

@app.post("/predictions", status_code=201)
def create_prediction(req: PredictRequest):
    """C : 예측 후 이력에 저장"""
    return store.create(req.features, run_model(req.features))


@app.get("/predictions")
def list_predictions(limit: int = 50):
    """R : 이력 목록 (최신순)"""
    return store.list_all(limit)


@app.get("/predictions/{pid}")
def get_prediction(pid: int):
    """R : 이력 단건"""
    row = store.get(pid)
    if row is None:
        raise HTTPException(status_code=404, detail="해당 예측 이력이 없습니다")
    return row


@app.put("/predictions/{pid}")
def update_prediction(pid: int, req: PredictRequest):
    """U : 입력을 바꿔 다시 예측하고 이력을 갱신"""
    row = store.update(pid, req.features, run_model(req.features))
    if row is None:
        raise HTTPException(status_code=404, detail="해당 예측 이력이 없습니다")
    return row


@app.delete("/predictions/{pid}")
def delete_prediction(pid: int):
    """D : 이력 삭제"""
    if not store.delete(pid):
        raise HTTPException(status_code=404, detail="해당 예측 이력이 없습니다")
    return {"deleted": pid}
코드 설명

status_code=201: 생성은 200이 아니라 201을 쓰는 게 REST 관행입니다.

{pid} + pid: int: URL의 경로 변수를 함수 인자로 받습니다. 타입을 int로 적으면 검증까지 자동입니다.

limit: int = 50: 쿼리 파라미터. 안 보내면 50.

raise HTTPException(404): PUT에서도 U 하기 전에 없으면 404입니다. store가 None을 주는 걸 그대로 404로 옮깁니다.

U가 단순 덮어쓰기가 아닌 것에 주목: run_model(req.features)를 다시 호출해 새 입력으로 재예측한 뒤 갱신합니다.

4예측하기 — /predict

4-1. 위험이 높은 상황 — 새벽 시간대 주거지 방화

curl -X POST http://localhost:8001/predict \
  -H "Content-Type: application/json" \
  -d '{"features":{"gu":24,"fire_type":0,"cause":5,"place_main":12,"place_sub":4,"month":3,"hour":6}}'
수행결과
{
  "prediction": 1,
  "label": "인명피해 있음",
  "proba": {
    "인명피해 없음": 0.0913,
    "인명피해 있음": 0.9087
  }
}
결과 설명

중랑구(24) · 방화(5) · 주거(12) · 공동주택(4) · 새벽 6시. 인명피해 확률 90.9%. 6단계 run_model이 만든 딕셔너리가 그대로 JSON이 된 것입니다.

4-2. 반대 상황 — 야외 쓰레기 화재

curl -X POST http://localhost:8001/predict \
  -H "Content-Type: application/json" \
  -d '{"features":{"gu":21,"fire_type":1,"cause":9,"place_main":1,"place_sub":22,"month":2,"hour":21}}'
수행결과
{
  "prediction": 0,
  "label": "인명피해 없음",
  "proba": {
    "인명피해 없음": 0.9887,
    "인명피해 있음": 0.0113
  }
}
결과 설명

같은 모델이 완전히 다른 답(98.9% 없음)을 냅니다. 두 요청을 반드시 모두 해보세요.

자주 하는 실수

선택 항목은 이름이 아니라 번호입니다. "gu": "중랑구"는 422 오류 — 4단계 PredictRequest가 걸러 냅니다.

7U — 수정 PUT /predictions/1

설명

1번(고위험)을 저위험 입력으로 바꿔 본다. 값만 바뀌는 게 아니라 다시 예측된다.

curl -X PUT http://localhost:8001/predictions/1 \
  -H "Content-Type: application/json" \
  -d '{"features":{"gu":21,"fire_type":1,"cause":9,"place_main":1,"place_sub":22,"month":2,"hour":21}}'
수행결과
{
  "id": 1,
  "features": {
    "gu": 21, "fire_type": 1, "cause": 9,
    "place_main": 1, "place_sub": 22, "month": 2, "hour": 21
  },
  "prediction": 0,
  "label": "인명피해 없음",
  "proba": { "인명피해 없음": 0.9887, "인명피해 있음": 0.0113 },

결과 설명

id는 1 그대로, label이 "있음" → "없음"으로 뒤집혔습니다. 8단계 update_prediction이 run_model(req.features)를 다시 호출했기 때문입니다.

created_at은 처음 만든 시각 그대로입니다 — 7단계 update의 UPDATE 문에 created_at이 없기 때문.
