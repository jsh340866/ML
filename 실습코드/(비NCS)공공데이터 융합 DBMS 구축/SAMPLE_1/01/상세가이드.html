01. 모델 학습 — 서울 화재 인명피해 예측

공공데이터포털의 소방청 화재 데이터로 "이 화재에 인명피해가 있을까"를 맞힌다. 학습한 모델을 02·03에서 서비스한다.

대상: 비전공 초보 · 소요: 약 2시간 · 데이터: data/소방청_화재발생정보_서울.csv (26,760건) · 산출물: ../02/model.pkl

워크플로우 — 로드 → EDA → 결측치 → 타겟분리·인코딩 → 분할(stratify) → 학습 → 평가(ROC-AUC) → 저장. 이 뼈대가 모든 이진분류에 그대로 반복됩니다.

데이터 출처: 공공데이터포털 > 소방청_화재발생 정보 (data.go.kr/data/15044003)  |  제공기관 소방청  |  파일 CSV  |  라이선스 이용허락범위 제한 없음

데이터셋 자세히 보기 — 컬럼 의미 · 출처 · 접근법 · 활용 방식

개요

전국 화재조사관이 국가화재정보시스템에 입력한 화재 조사 기록. 화재 1건 = 1행이다. 원본은 2020.01.01 ~ 2024.12.31 전국 191,510건이며, 이 실습은 서울특별시 26,760건만 추려 쓴다(시도 열은 값이 하나뿐이라 제외).

서울 26,760건 중 인명피해가 난 화재는 1,051건(3.9%) — 사망 발생 140건, 부상 발생 944건이다.

출처 · 받는 방법

공공데이터포털(data.go.kr) → 데이터 15044003 페이지 → 다운로드 버튼 (로그인 불필요). 받으면 파일명은 소방청_화재발생 정보_20241231.csv, 인코딩은 cp949, 크기 약 22MB.

이 폴더의 data/소방청_화재발생정보_서울.csv 는 그 원본에서 서울만 필터링해 둔 것(2.8MB)이라 바로 pd.read_csv 하면 된다.

점선 밑줄 용어에 마우스를 올리면 툴팁 설명이 나옵니다. 각 단계는 코드 → 수행결과 → 결과 설명 순서입니다.

이 단원의 진짜 주제 — 불균형 — 인명피해가 있는 화재는 전체의 3.9%뿐입니다. 이런 데이터에서 정확도는 착시를 일으킵니다. 그 함정을 눈으로 확인하고 대처법(class_weight)까지 익히는 것이 목표입니다.

핵심 개념 — 이진분류와 불균형

설명

이진분류: 정답이 두 가지(인명피해 있음 / 없음)인 문제. 모델은 특성(X)을 보고 정답(y)을 맞히는 규칙을 학습한다.

이 데이터의 정답은 직접 만든다. 인명피해 인원수가 들어 있으므로 1명이라도 있으면 1, 없으면 0으로 바꾼다.

비유: 소방서에서 "이 신고, 사람이 다칠 가능성이 큰가?"를 출동 전에 가늠하는 일. 다만 실제 화재의 96%는 인명피해가 없어서, "무조건 없다"고만 답해도 96점을 받는 불균형 함정이 있다.

해석 — 왜 새벽이 위험한가

새벽 5시(8.5%)가 저녁 7시(2.6%)의 3배 이상이다. 이유는 어렵지 않다 — 자고 있어서 늦게 알아차리고 대피가 늦다. 화재 건수 자체는 낮 시간대가 많지만, 다치는 비율은 새벽이 압도적이다.

겨울(12월·1월)에 화재도 많고 인명피해율도 높다 — 난방·화기 사용이 늘기 때문.

자치구는 건수와 위험도가 따로 논다. 강남구는 화재가 가장 많지만(2,024건) 인명피해율 상위는 동대문구·중랑구·도봉구다. "화재가 잦은 곳"과 "다치는 곳"은 다르다.

이 해석이 실습과 어떻게 연결되나

02·03에서 넣어 볼 두 입력이 바로 이 표들에서 나온 것이다.

고위험 = 건축,구조물(4.8%) + 방화(20.1%) + 주거·공동주택(5.3%) + 새벽 6시(5.8%) → 모델 예측 인명피해 있음 90.9%

저위험 = 기타 쓰레기 화재(0.7%) + 전기적 요인(2.7%) + 야외(0.6%) + 저녁 21시 → 모델 예측 인명피해 없음 98.9%

모델이 마법을 부리는 게 아니다. 이 표에 있는 규칙을 조합해 배운 것이다.

3전처리 — 결측치 처리

설명

날짜에서 월·시간대를 뽑는다(화재는 계절·시간대의 영향을 받는다). 그리고 사용할 열에 결측이 있는 행을 정리한다.

dt = pd.to_datetime(df['화재발생년원일'], errors='coerce')
df['월']     = dt.dt.month
df['시간대'] = dt.dt.hour

CATS = ['시군구', '화재유형', '발화요인대분류', '장소대분류', '장소중분류']
before = len(df)
df = df.dropna(subset=CATS + ['월', '시간대', TARGET])
print('결측 행 제거: %d행 -> %d행' % (before, len(df)))

결과 설명

0행 제거 = 정상입니다. 사용할 열에는 결측이 없기 때문(2단계에서 확인). dropna는 숫자 변환 실패 시를 대비한 안전장치입니다.

월·시간대는 원본에 없던 열을 만든 것입니다.

중급 포인트 — 누수 방지

여기선 채울 게 없었지만, 결측을 중앙값·최빈값으로 채운다면 전체 데이터가 아니라 학습 데이터(train) 기준으로 채워야 합니다. 전체 기준으로 채우면 검증 데이터 정보가 학습에 새어 들어가(데이터 누수) 성능이 실제보다 좋아 보입니다.

# 코드 열 이름은 영문으로. 이 이름이 그대로 API 의 JSON 키가 된다.

OPTIONS = {}

df['month'] = df['월']

FEATURES = [

중급 포인트 — 무엇을 입력에서 빼야 하나

사망·부상·인명피해(명)소계는 타겟 그 자체입니다. 입력에 넣으면 정답 보고 정답 맞히기라 반드시 뺍니다.

재산피해도 화재 조사가 끝나야 나오는 사후 결과값이라 뺐습니다.

남긴 7개는 화재 상황에서 바로 알 수 있는 정보뿐입니다 — 자치구·화재유형·발화요인·장소(대/중)·월·시간대.

중급 포인트 — 왜 코드 열 이름을 영문으로 하나

이 이름이 그대로 API 의 JSON 키가 됩니다(02). 한글 키로 만들면 윈도우 터미널에서 curl 로 보낼 때 인코딩이 깨져 {"detail":"There was an error parsing the body"} 가 납니다.

화면에 보이는 이름은 label(한글)이 따로 담당합니다. 기계가 쓰는 이름은 영문, 사람이 보는 이름은 한글 — 실무의 기본 원칙입니다.

왜 원핫 인코딩이 아닌가

교과서 워크플로우는 원핫 인코딩(get_dummies)을 씁니다. 여기서 정수 코드를 쓰는 이유는 두 가지입니다.

1) 서빙 계약: 02의 FastAPI가 feature_spec.json의 options 인덱스 번호를 그대로 받아 예측합니다. 원핫이면 열이 100개 넘게 흩어져 화면·API가 복잡해집니다.
2) 트리 모델: RandomForest는 정수 코드로도 구간을 잘 나눕니다(선형모델이라면 원핫이 필요).

핵심: 학습할 때의 코드 순서와 서빙할 때의 순서가 반드시 같아야 한다. 그래서 OPTIONS를 그대로 feature_spec.json에 저장한다(8단계).

7평가 — 정확도 vs ROC-AUC

설명

predict는 0/1, predict_proba(...)[:, 1]는 양성일 확률. 불균형이라 정확도만 보면 안 된다.

pred  = model.predict(X_te)
proba = model.predict_proba(X_te)[:, 1]      # 양성(인명피해 있음) 확률 열
acc  = accuracy_score(y_te, pred)
auc  = roc_auc_score(y_te, proba)
base = 1 - y_te.mean()                       # 전부 '없음'으로만 찍었을 때
print('정확도(Accuracy):', round(acc, 4))
print('  전부 "없음"으로 찍기만 해도 :', round(base, 4))
print('ROC-AUC        :', round(auc, 4))
print(classification_report(y_te, pred, target_names=LABELS, zero_division=0))

8모델 저장

설명

모델과 입력 정의를 02 폴더에 저장한다. feature_spec.json이 02의 API와 03의 화면을 자동으로 만들어 준다.

TITLE = '서울 화재 인명피해 예측'
DESC  = '자치구·화재유형·발화요인·장소·시기를 입력하면 해당 화재에 인명피해가 발생할지 예측합니다.'
for f in FEATURES:                 # 수치형 입력 범위 자동 계산
    if f['kind'] == 'num':
        s = X[f['name']]
        f.update(min=int(np.floor(s.min())), max=int(np.ceil(s.max())), default=int(round(s.median())))
spec = {'title': TITLE, 'description': DESC, 'target_labels': LABELS,
        'accuracy': round(float(acc), 4), 'roc_auc': round(float(auc), 4), 'features': FEATURES}
joblib.dump(model, '../02/model.pkl')
json.dump(spec, open('../02/feature_spec.json', 'w', encoding='utf-8'), ensure_ascii=False, indent=2)
print('저장 완료: ../02/model.pkl, ../02/feature_spec.json')
